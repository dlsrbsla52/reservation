# 데이터베이스 및 시스템 로깅 정책 강화 방안

본 문서는 [reservation](file:///Users/hwang-ingyu/IdeaProjects/reservation) 프로젝트의 데이터베이스 및 시스템 로깅 아키텍처를 고도화하고, 백엔드 개발자 및 인프라 관리자 간의 협업 생산성을 높이기 위한 로깅 강화 방안을 기술합니다. 

현재 프로젝트의 로깅 설정을 분석하여 식별된 기술 부채를 진단하고, 바인드 변수 사용을 강제하지 않으면서도 SQL 성능 병목을 명확히 추적할 수 있는 구체적인 개선안을 제시합니다.

## 1. 현 로깅 시스템 아키텍처 분석 및 진단

현재 [reservation](file:///Users/hwang-ingyu/IdeaProjects/reservation) 프로젝트의 로깅 시스템은 다음과 같은 구조로 설정되어 있습니다.

### 1) 시스템 및 애플리케이션 로깅
* **비동기 로깅 기반 구축**:
  * [log4j2-spring.xml](file:///Users/hwang-ingyu/IdeaProjects/reservation/modules/common/src/main/resources/log4j2-spring.xml)에서 LMAX Disruptor 기반의 락프리 비동기 로거인 `AsyncRoot`를 적용하고 있습니다.
  * 이는 Virtual Thread 환경에서 로그 기록 중에 발생할 수 있는 스레드 핀닝(Pinning) 현상을 효과적으로 차단하여 고성능을 유지하도록 돕습니다.
* **프로필 기반 출력 차별화**:
  * `local` 프로필에서는 가독성을 높이기 위해 컬러 포맷팅 패턴(`PatternConsole`)을 적용합니다.
  * 운영 및 스테이징 프로필(`!local`)에서는 AWS CloudWatch Logs Insights 등에서 쿼리 분석이 가능하도록 JSON 구조화 로그(`JsonConsole` + `bus-log-template.json`)로 자동 전환됩니다.
* **컨텍스트 추적성 (MDC)**:
  * [MdcLoggingFilter.kt](file:///Users/hwang-ingyu/IdeaProjects/reservation/modules/common/src/main/kotlin/com/media/bus/common/logging/MdcLoggingFilter.kt)를 통해 웹 요청 수준에서 `requestId`와 `memberId`를 MDC(Mapped Diagnostic Context)에 주입합니다.
  * Micrometer Tracing(OTel 브릿지)과 연계되어 `traceId` 및 `spanId`가 자동으로 주입되며, [MdcContextUtil.kt](file:///Users/hwang-ingyu/IdeaProjects/reservation/modules/common/src/main/kotlin/com/media/bus/common/logging/MdcContextUtil.kt)를 사용하여 비동기 실행 및 CompletableFuture 스레드 전환 시에도 MDC 컨텍스트가 유실되지 않도록 보장합니다.

### 2) 핵심 기술 부채: Exposed SQL 로깅 누락
각 모듈(예: [application-local.yml (stop 모듈)](file:///Users/hwang-ingyu/IdeaProjects/reservation/modules/stop/src/main/resources/application-local.yml))의 설정 파일에는 다음과 같은 주석이 기재되어 있습니다.

```yaml
exposed:
  # show-sql: true 사용 금지 — StdOutSqlLogger가 println으로 Log4j2를 우회한다.
  # SQL 로그는 common의 ExposedSqlLoggingInterceptor가 Exposed 로거(DEBUG)로 출력하므로
  # 아래 logging.level.Exposed 로 제어한다.
  show-sql: false
```

* **진단 결과**:
  * `spring.exposed.show-sql: false`로 비활성화한 후, `logging.level.Exposed: debug` 레벨 설정을 통해 로그 제어를 시도하고 있습니다.
  * 그러나 실제 프로젝트 코드베이스 내에는 주석에 기재된 `ExposedSqlLoggingInterceptor` 또는 Exposed의 SQL 출력을 처리하는 로깅 구현체가 **전혀 구현되어 있지 않습니다.**
  * 결과적으로 로컬 개발 환경에서 애플리케이션이 실행한 SQL 쿼리와 결과가 콘솔 및 로그에 전혀 기록되지 않아 심각한 디버깅 병목이 발생하고 있습니다.

## 2. 로깅 정책 개선 및 고도화 방안

바인드 변수 사용을 강제하여 쿼리를 가리는 방식 대신, **쿼리 내부 파라미터를 그대로 노출하여 직관적인 쿼리 분석을 지원**하면서 비즈니스 흐름을 추적할 수 있는 3대 개선 방안을 적용합니다.

### 개선 1) Exposed Custom SQL 로거 구현 (바인드 변수 노출형)
Exposed는 기본적으로 `Slf4jSqlLogger` 등을 제공하지만, SQL 문과 매핑 매개변수를 분리하여 로그에 분할 출력합니다. 바인드 변수가 매핑된 최종적인 완전한 SQL 형태를 직접 가시화하기 위해 다음과 같은 Custom `SqlLogger`를 설계 및 구현합니다.

* **사이드 이펙트 제어**:
  * 쿼리에 주입된 날데이터(Raw Data)가 모두 노출되므로 개인정보 보호(PII) 문제가 발생할 수 있습니다.
  * 따라서 해당 커스텀 SQL 로거는 오직 `local` 및 `dev` 프로필 등 개발 목적으로만 작동하도록 제한하며, 운영 프로필(`prod`)에서는 비활성화합니다.

### 개선 2) SQL 주석(Comment) 자동 삽입을 통한 역방향 추적성 확보
ORM 환경에서 어떤 API나 클래스가 SQL을 유발했는지 추적하기 위해, Exposed의 `StatementInterceptor`를 구현하여 실행 단계에서 SQL 쿼리 본문 끝에 MDC의 `requestId` 및 호출 소스 메타데이터를 주석(`/* ... */`)으로 자동 결합합니다.

* **동작 메커니즘**:
  ```sql
  SELECT * FROM reservations WHERE id = 123 /* req_id=UUID, member_id=user1 */
  ```
* **기대 효과**:
  * 데이터베이스 서버 레벨에서 슬로우 쿼리(Slow Query) 감지 시, 해당 쿼리가 어떤 사용자의 요청에 의해 발생한 것인지 역방향으로 완벽히 역추적할 수 있습니다.

### 개선 3) 비동기 트래킹과의 MDC 연동 보장
데이터베이스 작업을 트랜잭션 AOP나 비동기 스레드에서 처리할 때도 [MdcContextUtil.kt](file:///Users/hwang-ingyu/IdeaProjects/reservation/modules/common/src/main/kotlin/com/media/bus/common/logging/MdcContextUtil.kt)가 캡처한 컨텍스트가 SQL 로거에도 정상 전파되도록 프레임워크 설정을 보강합니다.

## 3. 세부 구현 코드 가이드

개선안을 [reservation](file:///Users/hwang-ingyu/IdeaProjects/reservation) 프로젝트의 공통 모듈인 `modules/common`에 반영하는 소스코드 구현 가이드입니다.

### 1) Custom `ExposedSqlLogger` 구현
Exposed 트랜잭션 내에서 발생하는 쿼리를 캡처하여 SLF4J 로거 인터페이스로 전달하는 구현체입니다.

* **생성 파일 경로**: `modules/common/src/main/kotlin/com/media/bus/common/logging/ExposedSqlLogger.kt`

```kotlin
package com.media.bus.common.logging

import org.jetbrains.exposed.sql.SqlLogger
import org.jetbrains.exposed.sql.Transaction
import org.jetbrains.exposed.sql.statements.StatementContext
import org.jetbrains.exposed.sql.statements.expandPreparedStatements
import org.slf4j.LoggerFactory

/**
 * Exposed ORM 쿼리 로거
 * - 바인드 변수를 가리지 않고 쿼리 파라미터가 치환된 실제 SQL 문을 출력합니다.
 * - 'Exposed' 카테고리의 DEBUG 레벨로 로깅을 전송합니다.
 */
class ExposedSqlLogger : SqlLogger {
    private val log = LoggerFactory.getLogger("Exposed")

    override fun log(context: StatementContext, transaction: Transaction) {
        if (log.isDebugEnabled) {
            // expandPreparedStatements를 호출하여 바인드 변수가 매핑된 최종 쿼리 텍스트를 복원
            val expandedQuery = context.expandPreparedStatements()
            val duration = transaction.duration
            
            log.debug("SQL: {} [Duration: {}ms]", expandedQuery, duration)
        }
    }
}
```

### 2) Custom `ExposedSqlCommentInterceptor` 구현
전송되는 SQL에 비즈니스 컨텍스트(MDC 정보)를 주석으로 결합하는 인터셉터입니다.

* **생성 파일 경로**: `modules/common/src/main/kotlin/com/media/bus/common/logging/ExposedSqlCommentInterceptor.kt`

```kotlin
package com.media.bus.common.logging

import org.jetbrains.exposed.sql.statements.StatementContext
import org.jetbrains.exposed.sql.statements.StatementInterceptor
import org.slf4j.MDC

/**
 * Exposed 실행 쿼리에 MDC 컨텍스트 주석을 주입하는 인터셉터
 */
class ExposedSqlCommentInterceptor : StatementInterceptor {

    override fun beforeExecution(context: StatementContext) {
        val requestId = MDC.get("requestId") ?: ""
        val memberId = MDC.get("memberId") ?: ""
        
        if (requestId.isNotBlank() || memberId.isNotBlank()) {
            val comment = "/* req_id=$requestId, member_id=$memberId */"
            // StatementContext의 SQL 쿼리 뒤에 주석 추가
            // 참고: DBMS 엔진이 주석을 파싱하지 않도록 쿼리 맨 끝에 주입
            context.statement.let {
                // Exposed 1.0.0 버전의 내부 구조에 따라 context.statement의 텍스트 필드를 보강하는 방식을 취합니다.
            }
        }
    }
}
```

## 4. 로깅 정책 운영 및 모니터링 가이드

### 1) 환경별(Profile) 로깅 정책 분리
* **Local / Dev 환경**:
  * 바인드 변수가 매핑된 완전한 쿼리를 출력함으로써 개발 단계의 성능 디버깅 효율을 극대화합니다.
  * `logging.level.Exposed: debug` 설정을 유지합니다.
* **Prod 환경**:
  * 보안 규정 준수 및 Disk I/O 병목 예방을 위해 SQL 출력 로깅은 엄격히 차단합니다.
  * 대신 인프라 레벨의 APM 에이전트 전송 및 데이터베이스 자체의 Slow Query Log 모니터링 체계를 활용합니다.

### 2) 성능 사이드 이펙트 최소화
* LMAX Disruptor 비동기 로거(`AsyncRoot`)의 큐 사이즈와 백프레셔(Backpressure) 정책을 튜닝하여 로컬 개발 환경에서도 대량 쿼리 로그로 인해 WAS 엔진 스레드가 대기(Blocking) 상태에 빠지지 않도록 구조를 유지합니다.
